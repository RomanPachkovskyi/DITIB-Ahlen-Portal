# LEGACY_IMPORT.md — План інтеграції старих даних

## Контекст

Є стара база членів громади у форматі Excel, а не у вигляді окремої SQL-бази.

Відомо:

- записів: **743**
- старий номер: у діапазоні **`0000001001` → `0000005299`**
- є багато пропущених номерів
- це означає, що історично номер **не перевикористовувався**: після видалення запису його номер також зникав
- у файлі є лише такі поля:
  - `номер`
  - `ім'я`
  - `прізвище`
  - `дата народження`
  - `адреса`
  - `індекс`
  - `місто`
  - `Anrede`
  - `телефон`
  - `мобільний`

Поточна система Laravel працює інакше:

- нові записи отримують `member_number` у форматі `DA-YYYY-NNNN`
- у новій системі є значно більше полів, ніж у старому Excel

---

## Прийняте рішення

### 1. Не змішувати старий номер з новим `member_number`

Старий номер з Excel **не повинен замінювати** поле `member_number` у новій системі.

Причини:

- стара нумерація має іншу бізнес-логіку
- у ній є великі пропуски через історичні видалення
- нова система вже має власний формат `DA-YYYY-NNNN`
- змішування двох систем номерів у одному полі ускладнить підтримку, пошук, листи, адмінку та майбутню автоматизацію

### 2. Старий номер зберігати окремо як історичний ідентифікатор

Рекомендовано додати для імпорту окреме поле:

- `legacy_member_number` — nullable, unique

Це поле потрібне тільки для:

- звірки з Excel
- пошуку старого члена по історичному номеру
- перевірки дублювань під час імпорту
- можливого друку/експорту для внутрішнього користування

### 3. Нові записи після запуску порталу повинні й далі отримувати тільки новий номер

Отже:

- імпортовані старі записи матимуть:
  - `legacy_member_number`
  - новий системний `member_number`
- усі нові заявки з форми матимуть:
  - тільки `member_number`
  - без `legacy_member_number`

## Рекомендована бізнес-логіка імпорту

### Імена

У старому Excel ім'я та прізвище лежать окремо, а в новій системі зараз є лише `full_name`.

На першому етапі імпорту достатньо:

- скласти `full_name = trim(vorname + ' ' + nachname)`

Але стратегічно правильніше з часом перейти на окремі поля:

- `first_name`
- `last_name`

Це дасть кращий пошук, сортування, генерацію листів і точнішу роботу з Filament.

### Телефони

У старому Excel є два номери:

- звичайний телефон
- мобільний

У поточній системі є лише одне поле `phone`.

Тому рекомендоване рішення:

- `phone` використовувати для **мобільного** номера
- для стаціонарного номера додати окреме nullable поле:
  - `phone_landline`

Якщо окреме поле не додавати, частина старих даних буде втрачена.

### Статус

Старі записи не є "новими заявками з сайту", тому їх не варто імпортувати як звичайний `pending`.

Рекомендовано:

- імпортовані історичні записи отримують `status = active`, якщо немає причин вважати інакше

Якщо є сумніви в актуальності конкретних людей, можна окремо розглянути:

- `inactive` для непідтверджених або застарілих записів

### Email

У старому Excel email немає.

Тому на момент імпорту:

- `email` краще зробити nullable у процедурі legacy-імпорту або заповнювати технічним шаблоном тільки якщо це реально потрібно

Рекомендація:

- **не вигадувати фальшиві email-адреси**, якщо без цього можна обійтись

### Фінансові та DSGVO-поля

У старому Excel немає підтвердженої SEPA/DSGVO-інформації.

Тому імпорт не повинен створювати фіктивну згоду.

Рекомендовано для старих записів:

- `zahlungsart = barzahlung` як безпечний дефолт, якщо спосіб оплати невідомий
- `sepa_zustimmung = false`
- `dsgvo_zustimmung = false`
- `zustimmung_at = null`
- `iban = null`
- `bic = null`
- `kontoinhaber = null`
- `kreditinstitut = null`

## Що потрібно змінити в схемі перед імпортом

Перед реальним імпортом рекомендовано підготувати мінімальні зміни БД:

1. Додати `legacy_member_number`
2. Додати `phone_landline`
3. Переглянути, чи може `email` бути nullable для старих записів
4. За потреби додати технічний прапорець:
   - `is_legacy_import`
   - або `import_source` зі значенням `excel_legacy`

Опціонально, але бажано в майбутньому:

5. Розділити `full_name` на:
   - `first_name`
   - `last_name`

## Рекомендований порядок робіт

### Етап 1 — Підготовка схеми

- додати нові поля для legacy-імпорту
- адаптувати валідацію моделі та адмін-форми
- не ламати поточну реєстрацію через сайт

### Етап 2 — Підготовка Excel

- нормалізувати колонки
- перевірити формат дат
- очистити пробіли, порожні значення, дублікати
- привести телефони до єдиного формату
- перевірити, чи немає дублів по комбінації:
  - ім'я
  - прізвище
  - дата народження

### Етап 3 — Dry run імпорту

- зробити artisan-команду імпорту з Excel/CSV
- спочатку прогнати на копії локальної SQLite
- згенерувати звіт:
  - скільки імпортовано
  - скільки пропущено
  - які рядки мають помилки
  - які можливі дублікати знайдено

### Етап 4 — Ручна перевірка

- перевірити в адмінці кілька десятків записів
- переконатися, що старий номер шукається
- переконатися, що телефони розкладені правильно
- переконатися, що нова форма реєстрації продовжує працювати окремо

### Етап 5 — Production import

- тільки після локального dry run
- окремо з резервною копією production-бази

## Висновок

Для цього проекту правильне рішення таке:

- старий Excel імпортуємо
- старий номер не змішуємо з `member_number`
- старий номер тримаємо окремо як `legacy_member_number`
- мобільний номер зберігаємо в `phone`
- звичайний номер зберігаємо окремо в `phone_landline`
- відсутні поля не вигадуємо, а імпортуємо з безпечними nullable/defaut значеннями

Це дозволить:

- не зламати поточну логіку порталу
- зберегти історичну прив'язку до старої таблиці
- підготувати акуратний імпорт 743 записів без втрати важливих даних
